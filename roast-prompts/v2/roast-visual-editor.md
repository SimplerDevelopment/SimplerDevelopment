# Roast V2: Visual Editor — SimplerDevelopment

**How to use:** Run `/roast` and feed it this brief (it has enough context to skip the clarifying questions). The council judges **two lenses**: (A) does this module earn its place inside the all-in-one platform, and (B) could it stand alone as its own product.

## The idea

SimplerDevelopment's Visual Editor is a block-based WYSIWYG page builder embedded in the portal: the portal shell renders around a sandboxed `<iframe>` that loads the live public-site renderer, and the two sides communicate exclusively through a typed postMessage protocol (30 message types). The portal shell owns block state and persistence; the iframe owns layout, selection hit-testing, drag-and-drop, and inline text editing. It supports multi-breakpoint viewport switching, an undo stack with coalescing (for drag and slider sessions), right-click context menus, a layers panel for block-tree navigation, per-block settings panels in the sidebar, element-level style editing (typography, spacing, background), real-time collaboration (presence avatars, live cursors via Y.Doc), and revision history.

The load-bearing differentiator is not the WYSIWYG canvas, and not the MCP block-write path alone — Webflow MCP server, Builder.io CMS MCP, Sanity Agent Actions, and Framer 3.0 all shipped AI-agent page composition over MCP by June 2026. The genuine differentiator is context density: because the visual editor shares a tenancy with CRM contacts, Company Brain (RAG over the client's own knowledge), automations, and Stripe billing, an AI agent — or a human working in the editor — can act on page layout, customer segments, brand voice, and subscription state in a single session without a data hand-off or format translation. The visual editor is the human review layer that wraps the agent, and the agent has the full business context, not just the page.

## Who it's for & how it makes money

- **Primary buyer:** Digital agencies using SD to build and manage client websites; their team members and clients who edit page content directly in the portal.
- **End user:** Agency staff and, potentially, their clients doing self-serve content edits in a white-labeled portal at the agency's own domain.
- **Monetization:** Bundled inside the "Sites" module subscription — not a standalone paid add-on today, and no standalone ambition is being chased. Value is retention and switching-cost, not a separate revenue line. A white-label SDK is a documented later option, not present-tense scope.

## The edge

- **Cross-domain bundle lock-in is the primary wedge.** The visual editor is co-located in one tenancy with CRM contacts, Company Brain (RAG over the client's own knowledge base), automations, and Stripe billing. An AI agent — or a human working in the editor — can act on page layout, customer segments, brand voice, and subscription state in a single session without a data hand-off or format translation. No standalone editor (Webflow, Framer, Builder.io, Plasmic) can replicate this without building the rest of the suite. **The one genuine co-location competitor is GoHighLevel**, which also shares CRM + site builder + automations + a knowledge base on one tenancy (its multi-client agency tier; the knowledge base is a separate add-on, so a like-for-like co-located suite is materially more than GHL's entry price); SD differentiates on depth, not the existence of co-location — a Company-Brain RAG layer (not a flat KB) and a live-renderer editing surface GHL's builder does not match. This is a retention and switching-cost play, not just a feature.
- **iframe-isolation delivers pixel-accurate preview — a correctness property, not a unique moat.** By loading the actual public renderer in the iframe, the editor shows a live preview structurally identical to what publishes, eliminating the "looks different in preview vs. published" class of bugs (the design-system-divergence problem that hits separate design-time renderers like Plasmic's). Builder.io's visual editor uses the same live-frontend iframe + postMessage architecture, so this is shared best-practice, not an unreplicated edge — it raises the quality bar and supports the bundle story, but it is **not** a standalone differentiator and should not be pitched as one.
- **MCP block-write path is now table-stakes, not the wedge.** The `posts_update` MCP tool + `blocks://schema` resource means an AI agent can compose or restructure an entire page in a single call — no export step, no copy-paste. Webflow MCP server, Builder.io CMS MCP, Sanity Agent Actions, and Framer 3.0 all ship equivalent AI-agent page composition over MCP as of June 2026. The MCP path is a cost-of-entry capability; the unbundleable capability is the cross-domain context the agent operates with.
- **Revision history and collaboration are bundled at no premium.** Y.Doc presence, live cursors, and revision history are first-class features included in the subscription. This is table-stakes, not a differentiator — Webflow made real-time collaboration free on all plans in Feb 2026 — but bundling it at no extra charge keeps the cost-absorption story intact for agency teams editing client sites collaboratively.
- **Undo coalescing for drag and slider interactions.** The `coalesce` flag on `BLOCKS_UPDATE` ensures drag-and-drop and slider scrubbing do not pollute the undo stack — a UX detail that distinguishes mature editors from shallow ones.
- **Lockstep block registration prevents schema drift.** The `simplerdev-block-type` skill enforces that every new block type registers in TS types, block settings, the iframe registry, and the production renderer in a single commit. This discipline is what keeps the agent's schema contract stable across the full suite context.

## Constraints

- Solo founder / tiny team; SD is already a ~357k-line shipping monorepo (Next.js 16, Drizzle/Postgres+pgvector, Mastra agents), now open-sourcing.
- **Visual editor scope is frozen at "good enough for agencies."** The prior ambition to match Webflow control-by-control is dropped; new visual-editor features are added sparingly, against a now-stabilized schema.
- **The god-file split is DONE (not an open blocker).** `BlockContentEditor.tsx` was decomposed to 98 lines and `HtmlRenderEditor.tsx` to 488 (from ~2018/1694) across commits `eeec676d` and `ebc4b842` (Phases 2–5), with `_hooks/`, `_lib/`, and `_components/` populated and unit tests added. The block schema the AI-native path depends on is now stabilized post-refactor — so the AI-native wedge can be demoed cleanly today, not after a multi-week debt paydown. (Remaining hygiene watch-item, not a money/data gate: the 30-type postMessage protocol crossing the iframe boundary has no stated origin/CSP validation — close that before a formal security review.)
- The white-label SDK path is documented as a future option; it is not being built or pitched now.
- Time-to-first-dollar and maintainability by a tiny team both matter.

## Roast it on two lenses

1. **Earns its place in the suite?** Does the cross-domain bundle lock-in — page + CRM + Company Brain + automations + Stripe in one session — create real switching cost and retention, or is the canvas still a shallow me-too WYSIWYG that dilutes focus and adds maintenance drag now that MCP page composition is table-stakes across all major editors?
2. **Could it stand alone?** No standalone ambition — bundled retention layer. This lens should probe whether the bundled positioning is credible and sustainable, or whether the cross-domain context claim is actually thin because agencies don't exercise it enough in practice to feel the lock-in.

## Riskiest assumption to pressure-test

That the cross-domain bundle lock-in is a durable, agency-convincing wedge — specifically, that a demo in which an agent touches page layout, a CRM contact segment, Company Brain context, and a Stripe billing state in a single session produces genuine "would switch for this" reactions from 5 target agencies, rather than a polite "interesting" from buyers who in practice only use the editor for simple content edits and never exercise the full context. (The sharper cut, per the council: ask each agency whether *it* runs that cross-domain workflow as a managed service — lock-in lands on the buyer who signs the check — or whether only their end clients do, which attenuates the lock-in.) The MCP page-composition lead is gone (Webflow, Builder.io, Sanity, Framer all shipped it by June 2026) and the god-file refactor is already done — so the bet is now entirely on context density: if agencies don't exercise the CRM/Brain/billing integrations deeply enough to feel the lock-in, the visual editor is a quiet, competent bundled feature, not a wedge.

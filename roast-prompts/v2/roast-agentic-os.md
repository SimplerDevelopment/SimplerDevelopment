# Roast V2: Agentic OS — SimplerDevelopment

**How to use:** Run `/roast` and feed it this brief. The council judges **two lenses**: (A) does this module earn its place inside the all-in-one platform, and (B) could it stand alone as its own product.

## The idea

SimplerDevelopment's Agentic OS is the layer that makes the entire platform operable by AI agents. Its load-bearing claim is not tool count — it is structural, default-on human approval over AI content/output writes (CMS, email, proposals/decks) across a tenancy-safe cross-domain surface. The **MCP server** (`lib/mcp/`) exposes the full platform as a custom MCP server: any MCP-compatible agent (Claude Code, Cursor, Mastra workflows, a custom harness) can operate CRM, Brain/RAG, CMS, email, bookings, surveys, contracts, automations, billing, and projects on behalf of a tenant using only a portal API key. Every tool is scope-guarded and tenant-scoped at the handler level. The structural differentiator is the `require_cms_approval` flag: toggled on a portal API key, it routes **CMS, email, and proposal/deck** write tools through the pending-change queue before they land (a toggle on the key itself, not a workflow the agency configures). **Honest scope (verified against `lib/mcp/tools/`):** only those three surfaces are behind the pending-change queue today. Brain writes, the bulk of CRM writes (only ~3 of ~34 CRM tools are gated, and not the headline contacts/deals writes), and automations/billing/integrations writes are NOT queued; surveys and bookings use a draft-then-review-link pattern (the write lands immediately, then mints an approval link), which is not the same as the pending-change queue. The brief claims gating only where the code actually does it — and extending the queue across the rest is the committed coverage roadmap. The **Agentic OS admin dashboard** (`lib/agentic-os/`) catalogs 45 registered skills/crons with invariant rules and estimated runtimes. It is a developer-internal tool only — and permanently so by architecture: its executor spawns a local `claude -p` CLI process, which cannot run on Vercel serverless. It is not, and will not be, pitched as a customer-facing feature (see Constraints). The shipped product surface is the MCP server above.

The repositioned pitch: SD is the agency OS where AI content/output writes — CMS pages, email campaigns, and proposals/decks — are structurally human-gated and auditable by default, in one authenticated session, over a pre-built tenancy-safe agency tool surface.

## Who it's for & how it makes money

- **Primary buyer:** Digital agencies who want to deploy AI agents on real business work — draft proposals, move deals, publish content, book clients, run Brain playbooks — with structural human approval gates, not just "AI summarizes emails."
- **End user:** The AI agent itself (Claude, a Mastra workflow, a custom harness) operating on behalf of the agency's team.
- **Monetization:** The MCP server is a moat feature bundled with the SD platform subscription — it drives retention by making the platform uniquely AI-operable. No standalone ambition. The optional "Agent Access" plan unlocks write-scope grants (`automations:write`, `brain:write`, `crm:write`) as a paid add-on tier — gateable once the cheapest test (below) confirms agencies price the approval gate as a buying reason, not a footnote.
- **Honest starting posture:** Agencies buy SD for CRM/Brain/CMS and discover agents after commitment. The agentic layer is a retention hook and a premium add-on wedge — not a purchase driver. GTM leads with the domains; the agentic layer is the reason agencies don't leave.

## The edge

- **Key-level, default-on approval gating over a pre-built cross-domain agency surface is the structural claim.** The `require_cms_approval` flag on a portal API key routes CMS, email, and proposal/deck writes through the pending-change queue as a platform primitive — default-on per key (the verified gated surface today; queue coverage of the remaining write tools is the roadmap). Agent gateways like Relay.app, Portkey, and Bifrost can gate generic API calls, but none ships a *pre-built, tenancy-safe agency tool surface* (CRM + CMS + email + decks + bookings in one session) with approval baked into the tool-authorization model. The defensible claim is the compound — gate × ready-made agency surface — not "nobody has an approval gate." This is the headline.
- **Cross-domain compound surface in one authenticated session.** A single portal API key gives a scoped agent access to CRM + Brain + CMS + contracts + automations in one session with no per-integration wiring. Competitors require agencies to wire their own tools against their own APIs — SD ships the surface pre-built, tenancy-safe, and scope-gated.
- **Scope grants make authorization granular and auditable.** Each portal API key carries an explicit `scopes` array. The scope gate runs before every tool call; denials are logged as agent-action audit rows. Agencies can grant an agent exactly `brain:read` + `crm:read` and nothing else — structural RBAC on the agentic layer that competitors treat as application-layer configuration, not platform primitive.
- **The approval gate is a shared primitive of the MCP write path — first-party AI coverage is still partial.** Third-party MCP agents hit the gate by default (`require_cms_approval` is `.default(true)` per key). For SD's own AI, coverage is honestly partial today: the portal chatbot *imports* the gate but only applies it when `AI_TOOL_APPROVALS_ENABLED=1` (env-gated, **default off** — labeled P0.3, in progress), and the internal Mastra Brain agent runs on its own surface and is **not** wired to the gate at all. So the brief claims a consistent guarantee only for the MCP-agent path; first-party gate coverage is on the roadmap, not shipped. (Three disclosed carve-outs to "default-on per key": editing an *unpublished fork* skips staging at `cms.ts:353` since nothing live is affected; the Brain Connect key-issuance flow currently hardcodes `requireCmsApproval:false` (being reconciled); and the **mobile** sign-in path (`mobile-sign-in/route.ts:163`, `mobile-auth/page.tsx:79`) hardcodes `requireCmsApproval:false` as a permanent-by-design exception for the first-party mobile client.)

## Constraints

- Solo founder / tiny team; SD is a ~357k-line shipping monorepo (Next.js 16, Drizzle/Postgres+pgvector, Mastra agents), now open-sourcing.
- The MCP server competes for mindshare with best-of-breed agent frameworks (LangChain/LangGraph, CrewAI, Relay.app, Lindy) that agencies may already pay for. SD's moat is not the framework — it is the pre-built, tenancy-safe, approval-gated tool surface those frameworks would have to build themselves.
- 400+ tool schemas are a maintenance liability as well as an asset: every Drizzle migration that changes a response shape is a silent tool-drift risk on a solo founder's watch. Pruning toward the three highest-value agency jobs (proposal drafting, deal movement, content publishing) de-risks this before it compounds.

**Scope — what is shipped vs. intentionally internal (no un-shipped product blocker):**

- **The product surface IS shipped and in production.** The MCP server (`lib/mcp/`) and the key-level `require_cms_approval` gate are live today — that is the entire agentic product, and it has no deferred blocker. The earlier "production-gate the dashboard" item is **withdrawn, not deferred**: the admin dashboard's executor literally `spawn`s a local `claude -p` CLI process, which cannot run on Vercel serverless at all. The dashboard is therefore **permanently internal local-dev tooling by architecture** (matching the prior product decision), never a customer-facing feature — so there is nothing to ship and nothing to pitch there. Ops-facing "observe/cancel a running agent" is a future, separate product that would require re-hosting execution on a persistent worker; it is explicitly out of scope and not claimed.
- **Positioning discipline (not a gate):** lead with "AI writes to your CMS, email, and proposals are human-approved by default, in one session," and with task-completion reliability on the highest-value jobs (proposal drafting, content publishing) — never with raw tool count. "400 tools" is not a go-to-market claim.
- **Cheapest test (no code, sequenced — does not gate GO):** before pricing the optional "Agent Access" write-scope add-on, ask 5–10 agencies whether they'd pay for AI agents that write to their CRM/CMS with every write human-approved. Leaning in → price it; shrug → it stays bundled retention only.

## Roast it on two lenses

1. **Earns its place in the suite?** Does the approval gate + scope-gated cross-domain surface create real retention and a credible premium add-on — or is it plumbing that agencies never consciously value and therefore never price?
2. **Could it stand alone?** No standalone ambition — this is a bundled retention layer and optional write-scope add-on only. The correct question for Lens B is: once the cheapest test lands, does the approval-gate primitive deserve its own pricing page within the SD bundle, or does it quietly carry the subscription without ever surfacing to the buyer?

## Riskiest assumption to pressure-test

That key-level approval gating is a *stated* buying reason — not just a reassurance agencies nod at and then forget. The prior brief's riskiest assumption (agencies choose SD *because* of MCP tools) is confirmed false. The de-risked version: the approval gate turns "discovered-after-purchase" into "reason not to leave," and eventually into a priced add-on for write-scope access. The cheapest test (Blocker C above) resolves this before any pricing decision. If agencies do not lean in on the gate, the agentic layer stays a quiet retention-infra investment, priorities shift to hardening the underlying domains, and tool-surface pruning becomes the primary near-term task.

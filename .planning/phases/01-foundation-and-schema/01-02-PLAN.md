## Threats and Mitigations

| Threat | Severity | Mitigation |
|--------|----------|------------|
| ALTER TABLE with NOT NULL on existing rows causes migration failure | HIGH | variant_id column is nullable (no NOT NULL constraint, no DEFAULT needed) — verified in research |
| Drizzle schema out of sync with SQL migration | HIGH | Write SQL first, then add matching Drizzle definitions; verify column names, types, and constraints match 1:1 |
| Running `drizzle-kit generate` produces unexpected diff output | MEDIUM | Do NOT run `drizzle-kit generate` — write SQL manually per project convention (drizzle/0041 pattern). Only run `drizzle-kit push` to apply. |
| Webhook secret stored in plain text | MEDIUM | secret column is varchar(64) — acceptable for HMAC signing keys which are not passwords. Future phases can encrypt if needed. |
| survey_webhooks.url allows arbitrary URLs (SSRF vector) | MEDIUM | Schema-level only; URL validation happens at the API layer in Phase 4 when webhook configuration UI is built |
</threat_model>

    ```sql
      "delay_hours" integer NOT NULL DEFAULT 0,
    ```

    If reordering is not safe, simply add `variantId` as a plain integer column without the `.references()` call, and let the SQL migration handle the foreign key:

**Component behavior per UI-SPEC states:**

State 1 — No condition: Show section label "Conditional Logic", sub-label "Show this field only when...", and "Add Condition" button.

State 2 — Single rule: Show one rule row with four controls:
1. Field selector `<select>` — lists eligible fields (filtered per D-03 to fields above/before current). Display: field label + field type icon. Value: field.id.
2. Operator selector `<select>` — four options per D-02:
   - "Is" (value: `equals`)
   - "Is not" (value: `not_equals`)
   - "Is one of" (value: `is_one_of`)
   - "Is not one of" (value: `is_not_one_of`)
3. Value input — **behavior depends on the selected operator:**
   - For `equals` / `not_equals`: single-value input. If the trigger field has options (select/radio/checkbox), render a `<select>` dropdown with that field's options. Otherwise render `<input type="text">` with placeholder "Enter value...".
   - For `is_one_of` / `is_not_one_of`: multi-value input. If the trigger field has options (select/radio/checkbox), render checkboxes for each option (allowing multiple selections). Otherwise render `<input type="text">` with placeholder "Enter values, comma-separated..." and parse comma-separated input into the `values: string[]` array on change (split by comma, trim whitespace).
4. "Add another rule" link below.

**IMPORTANT per D-02:** The `ShowIfRule.values` field is already `string[]`. For `equals` / `not_equals`, wrap the single value in an array: `values: [singleValue]`. For `is_one_of` / `is_not_one_of`, store all selected values directly: `values: ['A', 'B', 'C']`. The evaluator in Plan 01 already checks `values.includes(answer)` for equals and `!values.includes(answer)` for not_equals, so multi-value "is one of" maps to `equals` with multiple values, and "is not one of" maps to `not_equals` with multiple values. Specifically:
- UI operator "Is" -> stored as `{ operator: 'equals', values: [single] }`
- UI operator "Is not" -> stored as `{ operator: 'not_equals', values: [single] }`
- UI operator "Is one of" -> stored as `{ operator: 'equals', values: [multiple] }` (evaluator's `values.includes(answer)` checks if answer matches ANY value)
- UI operator "Is not one of" -> stored as `{ operator: 'not_equals', values: [multiple] }` (evaluator's `!values.includes(answer)` checks answer matches NONE)

State 3 — Compound (2+ rules): Same as State 2 plus:
- Each rule row has a remove button (Material Icons `remove_circle_outline`, `text-muted-foreground hover:text-destructive`).
- Per D-05: all rules are AND'd together. No AND/OR toggle in UI. The combinator is always 'AND'.
- Per D-06: unlimited rules, scrollable list.

**Data flow — converting between internal state and showIf shape:**

When the component mounts, parse `field.showIf`:
- If undefined: empty rules array
- If legacy shape `{ fieldId, values }`: convert to `[{ fieldId, operator: 'equals' as const, values }]`
- If compound shape `{ combinator, rules }`: use rules directly

When rules change, call `onChange({ showIf: { combinator: 'AND', rules } })`.
When all rules removed, call `onChange({ showIf: undefined })`.

Per D-05, always write compound shape `{ combinator: 'AND', rules: [...] }` for new rules.

**Smart value input:**
When the selected trigger field has a `type` of 'select', 'radio', or 'checkbox':
- For single-value operators (equals/not_equals): render a `<select>` dropdown with that field's `options` array.
- For multi-value operators (is_one_of/is_not_one_of): render a list of checkboxes, one per option, allowing multiple selections. Each checkbox label shows the option text. Selected options populate the `values` array.

When the trigger field has no options (text, number, etc.):
- For single-value operators: render `<input type="text">` with placeholder "Enter value...".
- For multi-value operators: render `<input type="text">` with placeholder "Enter values, comma-separated..." and split/trim on comma to produce the `values` array.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Admin builder -> showIf JSON | Admin configures rules via UI; saved as JSON in survey definition |
| Piping template -> DOM render | Token substitution output rendered in JSX — auto-escaped by React |
| Public form answers -> resolvePiping | Respondent answers flow into template substitution |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-05 | Injection (XSS) | resolvePiping in public form | mitigate | React JSX auto-escapes all string values. resolvePiping output is used as `{resolvedLabel}` in JSX, never via dangerouslySetInnerHTML. Acceptance criteria explicitly checks for absence of dangerouslySetInnerHTML with piped values |
| T-02-06 | Denial of Service | Unlimited rules per field (D-06) | accept | Rules are AND'd with Array.every() — O(n) linear evaluation. Even 100 rules would evaluate in microseconds. No recursion or exponential blowup possible |
| T-02-07 | Tampering | showIf rules targeting deleted fields | mitigate | deleteField clears previewAnswers entry. Evaluator returns false for undefined answers on 'equals' rules — field hides safely. Users can manually remove stale rules |
| T-02-08 | Information Disclosure | Field IDs exposed in builder header | accept | Field IDs are random 8-char strings with no PII. Exposed only to authenticated admin users in the builder |
</threat_model>

<objective>
Stand up a Google Cloud project configured for the Google Workspace integration: Internal-only OAuth consent screen, all required APIs enabled, OAuth client credentials issued, and a Pub/Sub topic + subscription for Gmail watch notifications. Capture all values into a reproducible runbook and into environment variables for local + production.

This is a **manual / non-autonomous** plan — a human must perform actions in console.cloud.google.com that cannot be scripted from this repo. The output is (a) live GCP infrastructure, (b) credentials in `.env.local` and Railway, and (c) a runbook documenting what was done so future engineers can reproduce or audit it.

Purpose: Provide the OAuth + Pub/Sub substrate every later phase depends on.
Output: GCP project state, env vars set, GCP-SETUP.md runbook committed.
</objective>

<threat_model>
## Threats and Mitigations

| Threat | Severity | Mitigation |
|--------|----------|------------|
| OAuth consent screen accidentally configured as External — exposes app to all Google users + triggers CASA audit requirement for restricted scopes | HIGH | Confirm "User Type: Internal" before adding any scopes. Re-verify before issuing credentials. Document the choice in GCP-SETUP.md. |
| OAuth client credentials committed to git | HIGH | Credentials only ever go to `.env.local` (gitignored) and Railway. `.env.example` documents only the variable names, never values. Pre-commit hook (existing) blocks .env files. |
| Restricted scopes (gmail.readonly) requested before consent screen is "Internal" verified | HIGH | Order of operations: (1) set User Type Internal, (2) verify, (3) add scopes. Never request restricted scopes until step 1 is locked. |
| Pub/Sub topic created without IAM grant for `gmail-api-push@system.gserviceaccount.com` — Gmail watch will fail silently | MEDIUM | Grant `roles/pubsub.publisher` on the topic to that exact service account immediately after topic creation. Verified in GCP-SETUP.md test step. |
| Push subscription endpoint URL hardcoded to dev/staging URL, breaks when promoted to prod | MEDIUM | Use environment-specific subscriptions (one per Railway env). Document in GCP-SETUP.md. |
| Wrong redirect URIs registered, OAuth callback fails with redirect_uri_mismatch | MEDIUM | Register all three: localhost dev, staging, prod. Document in GCP-SETUP.md. |
| GCP billing not enabled — Pub/Sub silently rejects publishes | MEDIUM | Setup runbook step 2 verifies billing account is linked before proceeding. |

</threat_model>

<existing_patterns>
- Existing Google integrations (`lib/google-calendar.ts`, `lib/google-website-oauth.ts`) use env vars `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET`. We use **distinct** env var names (`GOOGLE_WORKSPACE_CLIENT_ID` etc.) so the new flow is a separate OAuth client, isolating risk: if we revoke the new credentials, calendar booking continues working.
- `.env.example` is the source of truth for what env vars exist. Anything we add to local `.env.local` or Railway must also be added here as a key with no value.
- `wrangler.toml` patterns from `workers/email-inbound` are the precedent for the future `workers/google-webhook` — it will need `PUBSUB_VERIFICATION_TOKEN` to verify push notification authenticity.
</existing_patterns>

  <action>
    Verify the OAuth credentials actually work end-to-end before declaring this plan done. The future code in 01-03 will rely on this, and a typo in a redirect URI now would only surface mid-Phase-2.

    1. **Build an authorize URL manually** (paste in browser, replace `{CLIENT_ID}`):
       ```
       https://accounts.google.com/o/oauth2/v2/auth?
         client_id={CLIENT_ID}&
         redirect_uri=http://localhost:3000/api/portal/integrations/google/callback&
         response_type=code&
         scope=https://www.googleapis.com/auth/userinfo.email%20openid&
         access_type=offline&
         prompt=consent
       ```
       (URL-encode in one line; the example is wrapped for readability.)

    2. **Sign in with your Workspace account.** You should see the consent screen showing "SimplerDevelopment Portal wants access to..." with the requested scopes. Approve.

    3. **You'll be redirected to localhost.** The redirect will fail to load (no server running yet) BUT the URL will contain `?code={authorization code}`. That confirms the OAuth flow round-trips correctly.

    4. **Exchange the code with curl** (within 5 minutes — codes are single-use and short-lived):
       ```bash
       curl -s -X POST https://oauth2.googleapis.com/token \
         -d "code={CODE}" \
         -d "client_id={CLIENT_ID}" \
         -d "client_secret={CLIENT_SECRET}" \
         -d "redirect_uri=http://localhost:3000/api/portal/integrations/google/callback" \
         -d "grant_type=authorization_code"
       ```
       Response should be a JSON object with `access_token`, `refresh_token`, `expires_in`, `id_token`. **The presence of `refresh_token` is the key thing** — confirms `access_type=offline&prompt=consent` worked.

    5. Discard the tokens. They were just for verification. (Optionally revoke: `curl -X POST "https://oauth2.googleapis.com/revoke?token={ACCESS_TOKEN}"`)

    Append to GCP-SETUP.md:
    ```markdown
    ## Smoke Test

    - **Tested:** {date}
    - **Authorize URL:** ✓ consent screen renders with correct app name + scopes
    - **Code exchange:** ✓ refresh_token present in response
    - **Tester:** {your name}
    ```

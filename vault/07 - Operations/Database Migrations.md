# Database Migrations

## The cardinal rule

**Never hand-edit files in `drizzle/`.**

All `drizzle/*.sql` files are generated by `drizzle-kit`. Editing them manually creates drift between the schema source of truth (`lib/db/schema/`) and the migration journal (`drizzle/meta/`). If you need to correct a migration, edit the schema module and regenerate.

## Normal generate → apply flow

### 1. Edit the schema module

Schema lives in `lib/db/schema/` — one TypeScript file per domain area. Always import from `@/lib/db/schema` (the barrel at `lib/db/schema/index.ts`), never from a specific sub-module directly.

### 2. Generate the migration SQL

```bash
bun run db:generate
```

This runs `drizzle-kit generate` and emits a new `drizzle/<NNNN>_<description>.sql` file. Review the generated SQL before applying — check for unintended `DROP` statements on existing columns or indexes.

### 3. Apply to local / staging database

```bash
bun run db:migrate
```

This automatically runs `db:verify-target` first. If `DATABASE_URL` points to a production host, the command refuses and exits 1 (see Safety guard below).

### 4. Verify migration applied

```bash
bun run db:studio
```

Opens Drizzle Studio at `localhost:4983` — browse tables and confirm the new columns/tables exist.

## Safety guard: `db:verify-target`

`scripts/verify-db-target.ts` is run before every `db:migrate` and `db:push`. It reads `DATABASE_URL`, checks for known production proxy hostnames, and exits 1 if it detects prod without an explicit override.

Production indicators: `$PROD_DATABASE_URL` host patterns, `RAILWAY_ENVIRONMENT_NAME=production`.

**To override (intentional prod operation only):**

```bash
ALLOW_PROD=1 bun run db:migrate
```

Do not add new production hostnames to `.env` without updating the indicator list in `scripts/verify-db-target.ts`.

## Local test database

Integration tests need a real Postgres instance. Start one locally with:

```bash
bun run test:integration:local
```

This calls `scripts/start-local-db.sh` (requires `postgresql@17` via Homebrew: `brew install postgresql@17`), then runs the integration suite against `postgresql://$USER@localhost:5432/simplerdev_test`.

To start the DB without running tests:

```bash
./scripts/start-local-db.sh
```

Then point your `.env.local` at it:

```
DATABASE_URL=postgresql://<your-username>@localhost:5432/simplerdev_test
```

## Tenancy regression gate

After any data-access change (tables, queries, API routes), run:

```bash
bun run test:tenancy
```

This runs `scripts/test.sh --layer=integration --tag=tenancy --no-coverage`. It is the cross-tenant data-leak regression suite. Tenancy leaks have shipped before — this suite catches them. The pre-push hook also triggers it automatically when `lib/db/`, `app/api/`, or `lib/active-client.ts` are in the diff.

## Resolving migration conflicts

If two branches each added a migration and their sequence numbers collide:

1. Determine which branch is the base (usually the one already on staging).
2. On the branch being rebased, delete the conflicting `drizzle/<NNNN>_*.sql` and its `drizzle/meta/` snapshot.
3. Rebase onto the base branch.
4. Re-run `bun run db:generate` to emit a new migration with the next available sequence number.
5. Verify with `bun run db:migrate` on a local DB.

## Production migration caveat

> From `lib/db/CLAUDE.md`: The Drizzle migration tracker is currently out of sync with prod. `bun run db:migrate` against prod fails. Schema changes must be **hand-applied** against the production database. See memory `project_sd2026_drizzle_tracker_drift`.

Standard procedure until tracker drift is resolved:
1. Generate migration locally with `bun run db:generate`.
2. Review the `drizzle/<NNNN>_*.sql` file.
3. Connect to production Postgres directly and apply the SQL manually.
4. Confirm the schema change took effect before deploying the application code that depends on it.

Vercel deploy does NOT run migrations automatically — schema must be applied before deploying code that references new columns or tables.

## HNSW index on `brain_embeddings`

The `brain_embeddings` table is declared in `lib/db/schema/brain.ts` but its HNSW index is managed separately in `drizzle/0061_brain_embeddings.sql` (pgvector HNSW indexes cannot be reconciled by drizzle-kit). **Never run `drizzle-kit push --force` against a database that contains real brain embeddings** — it will silently drop the HNSW index, causing severe embedding query performance degradation with no error.

Use journaled `bun run db:migrate` for all production schema changes.

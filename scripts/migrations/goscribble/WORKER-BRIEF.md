# Scribble migration — worker brief (read first)

You are building one or more pages for the Scribble (goscribble.ai) site migration into the SimplerDevelopment platform. Pages are stored in `posts.content` (prod DB) as block JSON and rendered by the site renderer. **All pages are DRAFTS** (`published: false`) — never publish.

## Read these before writing anything
1. `scripts/migrations/goscribble/_brand.ts` — shared brand constants + block helpers (section, heading, text, overline, spacer, button, cardGrid, stats, accordion, upsertPost, GLASS_BTN_STYLE, BRAND, NAVY_GRADIENT). **Reuse these. Do not edit this file.**
2. `scripts/migrations/goscribble/import-home.ts` — the reference pattern. Mirror its structure exactly: build a `blocks` array, then `await upsertPost({...})`.
3. `scripts/migrations/goscribble/COLOR-MAP.md` — brand palette + the home color rhythm. Apply the SAME palette to your pages.

## Source content for your page(s)
- Exact prose (headings/paragraphs/lists/links): `scripts/migrations/goscribble/extracted/<page>.txt`
- Raw HTML (for stat/card values inside divs): `scripts/migrations/goscribble/extracted/html/gs-<page>.html`
- To pull values out of styled divs, run:
  `node scripts/migrations/goscribble/extracted/extract-classes.mjs scripts/migrations/goscribble/extracted/html/gs-<page>.html <class1> <class2> ...`
  (inspect the HTML for the relevant class names — e.g. stat-number, stat-label, card titles).
- If something is still unclear, WebFetch `https://goscribble.ai/<page>.html`.

## Block rules — follow EXACTLY (these are real renderer constraints)
- Every block needs `id` (unique string), `type`, `order`. The `section()` / helper functions handle order for you.
- `image` block → field is `url` (NOT src) + `alt`.
- `button` block → field is `url` (NOT link) + `text`. Button COLOR comes from branding; for a secondary button on a DARK section, pass `{ variant:'secondary', style: GLASS_BTN_STYLE }` so it's visible (white text). On light sections, `{ variant:'secondary' }` is navy-on-white (fine).
- NO `faq` block — use `accordion`.
- NO `logo-cloud` — use `cardGrid` (icon cards) for EHR/integration logos (we have no logo image files).
- `section`: use `style.*` for padding/bg (the helper does this); `maxWidth` is a direct prop (helper handles it). Always keep `paddingLeft/Right` ≥ 24px (helper default).
- For service/feature cards, use the `icon` prop (Material Icon name) NOT `image` — card images render as tall 192px strips. Pick sensible Material Icons.
- `text`/`heading` content may contain simple inline HTML (`<br>`, `<strong>`). Don't embed `<div>`/`<span style>` icon hacks.
- Content is serialized by `upsertPost` as `JSON.stringify({ blocks, version: '1.0' })` — just pass the blocks array.

## Design pattern per section (match the source's own light/dark sequence)
- Lead each section with `overline()` (teal eyebrow) → `heading(level 2)` → optional `text()` subtitle → content.
- Alternate section backgrounds to match the SOURCE page (most sections light `#F7F9FC` or white; use a DARK navy-gradient section for hero-like or CTA bands). Do NOT impose generic alternation — read the source.
- Every page should end with a CTA band (dark navy gradient `section({ gradient:true, dark:true })`) driving "Book a Demo" → `BRAND.demoUrl`, mirroring the home page's final CTA, UNLESS the page is legal (privacy/terms) — those are plain prose, no CTA band.
- Legal pages (privacy-policy, terms-of-service): a single light `section` (maxWidth ~760px) with a heading + the full prose as `text` blocks (preserve all clauses). No marketing flourish.
- Blog articles: a short DARK navy hero `section` (eyebrow "Article/Blog", H2 title, byline text) then a light `section` (maxWidth ~760px) with the article body as heading/text blocks in order. End with the standard CTA band.

## Page meta
Set `seoTitle` + `seoDescription` from the source page's `<title>` and meta description (in the .txt / html). Set `slug` to the page filename without `.html` (e.g. `for-agencies`, `article-pdgm`, `privacy-policy`). `title` = a clean human title. `postType: 'page'` for marketing/legal; `postType: 'blog'` for the 4 articles (also set `excerpt`).

## Run it
`bunx tsx scripts/migrations/goscribble/import-<slug>.ts` — confirm it prints `created post #N` or `updated post #N`. The websiteId is read from `data/ids.json` (prod, websiteId 3) exactly like import-home.ts. Do NOT modify data/ids.json. Do NOT run the dev server or publish — the boss handles visual QA centrally.

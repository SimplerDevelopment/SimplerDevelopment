# Contributing to SimplerDevelopment

Welcome, and thank you for considering a contribution. SimplerDevelopment is an open-source, MCP-native multi-tenant platform — a real production codebase that has been open-sourced rather than a demo project. We value **small, focused PRs** that move one thing forward clearly. A tight PR with a clear description ships faster and reviews better than a sprawling one.

If you are new here, start with a [`good first issue`](../../issues?q=label%3A%22good+first+issue%22) and read the [Getting started](#getting-started-local-dev) section below. Questions are welcome in issues.

---

## Getting started (local dev)

**Requirements:** Bun 1.3.11+, PostgreSQL 14+, Node.js 20+ (needed for a few `tsx` scripts).

```bash
# 1. Install dependencies (always use bun — never npm or yarn)
bun install

# 2. Copy the environment template
cp .env.example .env.local
```

Open `.env.local` and fill in the key values. The minimum set to get the app running:

| Variable | Purpose | How to generate |
|---|---|---|
| `DATABASE_URL` | Postgres connection string | Point at a local Postgres 14+ instance |
| `NEXTAUTH_SECRET` | NextAuth session signing | `openssl rand -hex 32` |
| `ENCRYPTION_KEY` | Encrypts stored client API keys (64 hex chars) | `openssl rand -hex 32` |
| `PORTAL_KMS_KEY` | Plugin JWT signing key (base64) | `openssl rand -base64 32` |
| `ANTHROPIC_API_KEY` | Platform-level AI fallback | Anthropic Console |
| `OPENAI_API_KEY` | Brain embeddings | OpenAI Console |

See the [README](README.md) for the full variable list. Optional integrations (Stripe, Resend, Google Workspace, AWS S3, Dropbox Sign) can be left empty for local work unless your change touches those areas.

**Postgres requirement:** the `vector` (pgvector) extension must be installed on your database. Most Postgres distributions support it; see the [pgvector docs](https://github.com/pgvector/pgvector) for installation instructions.

```bash
# 3. Apply database migrations
bun run db:migrate

# 4. (Optional) Seed an admin user for local testing
bun run db:seed

# 5. Start the dev server
bun dev
# → http://localhost:3000
```

## How the codebase is organized

This is a ~357k-line monorepo. It has three distinct route trees, each serving a different audience:

| Route prefix | Audience | What it does |
|---|---|---|
| `app/admin/**` | Internal / super-admin | Cross-tenant global administration |
| `app/portal/**` | Per-tenant client | The client-facing portal: websites, CRM, Brain, billing, etc. |
| `app/sites/**` and `app/s/**` | Public | Per-tenant public websites rendered from typed block content |

Business logic lives in `lib/` (organized by domain), shared React components in `components/`, and each test layer in `tests/`. Each major subsystem has a nested `CLAUDE.md` that documents its invariants, god-file warnings, and patterns — read the nearest one before making changes in that directory:

| Area | Start here |
|---|---|
| Portal pages / tenant UI | `app/portal/CLAUDE.md` |
| Per-tenant public sites | `app/sites/**`, `app/s/**` |
| Block registry and schemas | `lib/blocks/CLAUDE.md` |
| Visual block editor | `components/portal/visual-editor/CLAUDE.md` |
| Database schema and migrations | `lib/db/CLAUDE.md` |
| MCP server tools | `lib/mcp/CLAUDE.md` |
| AI / Company Brain / RAG | `lib/ai/CLAUDE.md` |
| Tests, coverage, gates | `tests/CLAUDE.md` |
| Auth and roles | `docs/guides/USER_MANAGEMENT.md` |
| Block editor JSON schema | `docs/guides/BLOCK_EDITOR_GUIDE.md` |

### Architecture invariants — please read these

Breaking any of the following will likely cause a failing gate or a data leak:

**Tenancy.** Every tenant-scoped table carries a `clientId` and/or `siteId` column. All queries must filter on these. Never derive tenant identity from a query parameter or request body — always resolve it from the session via `lib/active-client.ts` and the site-resolver middleware. Run `bun test:tenancy` after any data-access change.

**API envelope.** Every route handler under `app/api/portal/**` returns `{ success: true, data: ... }` or `{ success: false, error: "..." }`. Do not hand-roll this pattern — use `simplerdev-feature-scaffold` (described below) to generate routes correctly.

**Blocks are universal, never client-specific.** A block type must work for every tenant. Never write `if (clientId === ...)` logic inside a block. If you need per-client presentation differences, they belong in `lib/branding/` or a feature flag, not the block itself.

**Do not touch these files directly:**

| Path | Rule |
|---|---|
| `drizzle/*.sql` | Generated by `bun run db:generate`; never hand-edit |
| `bun.lock` | Updated only by `bun add` / `bun remove` |

## Common contributions, step by step

### Adding a new block type

A block is typed JSON stored in `posts.content`. Adding one correctly requires five coordinated changes: a TypeScript interface, a registry entry, a render component, a production renderer case, and the `/api/blocks` metadata. These must all move together — a missing piece leaves the editor broken or the block unrenderable.

**Use the `simplerdev-block-type` scaffolding skill** (Claude Code: `/simplerdev-block-type`) rather than adding these by hand. Hand-rolled blocks have historically missed at least one of the five steps.

Reference: `lib/blocks/CLAUDE.md` and `docs/guides/BLOCK_EDITOR_GUIDE.md`.

### Adding a new MCP tool

The platform exposes 200+ MCP tools via `lib/mcp/`. Each tool requires four things in lockstep: a handler function in `tools/<domain>.ts`, a Zod input schema, a scope guard (`hasScope`), and telemetry registration. Missing the scope guard is a tenancy/permission leak.

**Use the `simplerdev-mcp-tool` skill** to generate all four pieces together.

After adding a tool, the registry baseline test (`tests/unit/mcp-tool-registry-baseline.test.ts`) will fail until you update `EXPECTED_TOOLS` in that file. This is intentional — it prevents silent drift. Update the constant as part of your PR.

Reference: `lib/mcp/CLAUDE.md`.

### Adding a new CRUD resource

New resources (a new data entity with API routes and a portal UI) follow a lockstep pattern: Drizzle schema → API route (with `{ success, data | error }` envelope and tenant scoping) → e2e test. Use the `simplerdev-feature-scaffold` skill to generate the schema, route, and e2e scaffold together. For portal-only pages with no new API, use `simplerdev-ui-scaffold`.

Reference: `app/portal/CLAUDE.md` and `docs/guides/DATABASE.md`.

### Changing the database schema

1. Edit the relevant file in `lib/db/schema/` (one file per domain area, e.g. `brain.ts`, `crm.ts`, `cms.ts`). Import from `@/lib/db/schema` — never from a specific module file.
2. Run `bun run db:generate` to emit a new migration file under `drizzle/`.
3. Run `bun run db:migrate` to apply it locally. The migrate script refuses connections to known production URLs.
4. If the new table holds tenant data, add `clientId` + a tenancy test fixture in the same PR.

**Never hand-edit files under `drizzle/`.** They are generated and will be overwritten.

Reference: `lib/db/CLAUDE.md` and `docs/guides/DATABASE.md`.

### Changing AI behavior (Company Brain / portal chatbot)

Before making any Anthropic or OpenAI call in a new feature, you must call `resolveClientApiKey` (never read `ANTHROPIC_API_KEY` directly), check `checkAiPlanGate` (prevents billing the platform for starter-tier clients), and call `recordAiUsage` after the call (fire-and-forget). Tool results flowing to the model must pass through `sanitizeToolResult` to strip credentials and PII.

Coverage floor for `lib/ai/` is 70% — keep tests updated when you add code here.

Reference: `lib/ai/CLAUDE.md`.

## Testing

Tests are split into three layers under `tests/`:

| Layer | Path | Runner | What it covers |
|---|---|---|---|
| Unit | `tests/unit/` | Vitest | Pure functions, single components, schema validators |
| Integration | `tests/integration/` | Vitest + real DB | API routes, multi-table flows, tenancy regressions |
| E2E | `tests/e2e/` | Playwright | Golden-path user journeys, visual flows |

Integration tests must run against a real database — mocking the DB produces false confidence and has caused real regressions. Use `bun test:integration:local` to spin up a local test database automatically.

### Gate commands

```bash
# Unit tests (fast, no DB)
bun test

# Integration tests (spins up a local DB)
bun test:integration:local

# Full E2E suite
bun run test:e2e

# QA gate — run before declaring any feature done
bun test:critical

# Tenancy regression — REQUIRED after any data-access change
bun test:tenancy

# Lint and typecheck
bun run lint
bun run typecheck
```

`bun test:critical` runs the `@critical`-tagged Playwright golden-path subset and is the minimum QA gate before a PR is ready. `bun test:tenancy` is mandatory after any change to queries, schema, or data-access logic.

A local CI gate script runs the same checks that CI runs:

```bash
scripts/ci-local.sh           # lint + typecheck + unit tests
scripts/ci-local.sh --full    # + tenancy + critical E2E (needs DB + Playwright)
```

### Coverage floors

Project-wide: 60% lines/statements/functions. Higher floors apply to sensitive areas:
- `lib/billing`, `lib/ai`, `lib/agency`, `lib/esign`, `lib/chat`: **70%**
- `lib/crypto`: **90%**

See [`tests/CI-GATES.md`](tests/CI-GATES.md) for the full gate definitions and [`tests/TESTING_PLAN.md`](tests/TESTING_PLAN.md) for the test responsibility model.

## Pull request process

1. **Fork and branch.** Use `feat/<topic>` for new features, `fix/<topic>` for bug fixes. Target `main`.

2. **Conventional commits.** Commit messages follow the `type(scope): description` format. Common types: `feat`, `fix`, `chore`, `docs`, `ui`, `refactor`. Common scopes: `brain`, `crm`, `blocks`, `editor`, `mcp`, `billing`, `google-workspace`, `survey`, `workers`, `build`. Examples:
   ```
   feat(blocks): add testimonial-grid block type
   fix(crm): scope deal queries to clientId
   docs(mcp): update tool registry baseline comment
   ```

3. **Keep PRs focused.** One feature or fix per PR. A PR that mixes unrelated changes is harder to review and slower to merge.

4. **Run the gates before opening.** At minimum: `bun run typecheck`, `bun run lint`, `bun test`. If your change touches data access: `bun test:tenancy`. If your change touches a user-facing flow: `bun test:critical`.

5. **Write a useful PR description.** Include: what changed and why, how to test it locally, which gates you ran, and any gotchas or follow-up work. Screenshots help for UI changes.

Maintainers aim to review PRs promptly. If you have not heard back within a few days, feel free to ping in the issue thread.

## Code style

- **TypeScript throughout.** Match the typing style of the surrounding code — avoid `any` where a specific type is derivable.
- **Material Icons, not emojis,** in any rendered portal or public-facing UI. Icon names go in the `icon:` field of block definitions (e.g. `'title'`, `'image'`), not the rendered glyph.
- **No client-specific code paths in the platform.** If you find yourself writing `if (clientId === ...)` in shared code, that logic belongs in `lib/branding/` or a feature flag instead.
- Match the indentation, import order, and file structure of nearby files. ESLint enforces the stricter rules; the rest is convention.

## Reporting bugs and security issues

**Security vulnerabilities:** please do NOT open a public GitHub issue. Follow the private disclosure process in [SECURITY.md](SECURITY.md).

**Bugs and feature requests:** open a [GitHub issue](../../issues). Use the `bug` label for confirmed bugs and `enhancement` for feature ideas. If you are looking for something to work on, filter by [`good first issue`](../../issues?q=label%3A%22good+first+issue%22).

# Glossary — SimplerDevelopment

> Alphabetical definitions of domain terms used throughout the codebase and documentation. Written for developers and AI agents. Each entry cross-links to the doc that covers the topic in depth.

## A

### AB Test / Experiment
A server-side split test that assigns public visitors to variants using FNV-1a deterministic bucketing. The active model is polymorphic: supported targets are `post` and `deck`. Survey and email targets are typed in `lib/ab/access.ts` but return `null` (not wired). Deck AB is implemented but not called on public render paths. Results surface a two-proportion z-test significance score. See `app/portal/experiments/`.

### Admin (route tree)
The global internal agency panel at `app/admin/**`. Routes here are only accessible to agency-level super-admins, not to client users. Distinct from the client portal. See [architecture](./architecture-for-agents.md).

### Approval Link
A tokenized URL (`/approve/[token]/`) generated by MCP write tools before live-content mutations go public. A human clicks the link to review a WYSIWYG preview and approve or reject. Implemented in `lib/mcp/approvals.ts`. Metadata and draft operations bypass the approval step. MCP tools: `approvals_list`, `approvals_get`, `approvals_approve`, `approvals_reject`. See [tool reference](./tool-reference.md).

### Automation Rule
An event-driven, one-shot rule of the form `trigger → conditions → actions`. Created via a natural language bar or the rule editor. Distinct from the Visual Workflow — automation rules are stateless and one-shot; visual workflows are durable and multi-step. Managed via `automations_*` MCP tools and `app/portal/automations/`. See [workflow reference](./workflow-reference.md).

## B

### Block
The atomic unit of page content. A block is a JSON object with a `type` string and a typed `data` payload. All post content is a JSON tree `{ blocks: Block[], version: '1.0' }` stored in `posts.content`. Blocks are universal — the registry is shared across all tenants; no block type is client-specific. 47+ built-in types registered in `lib/blocks/registry.ts`. See [tool reference](./tool-reference.md) and `docs/guides/BLOCK_EDITOR_GUIDE.md`.

### Block Registry
The central manifest at `lib/blocks/registry.ts` that lists every block type with its schema, display name, category, and flags (e.g. `emailOnly`). Adding a new block type requires lockstep changes across five files; use the `simplerdev-block-type` skill rather than hand-rolling.

### Block Template
A saved, reusable block configuration stored in the database and surfaced in the block picker. Distinct from a post: a block template is a single block (or group) saved for reuse, not a full page. MCP tools: `block_templates_list`, `block_templates_get`, `block_templates_create`, `block_templates_update`, `block_templates_delete`, `block_templates_fork`, `block_templates_publish`.

### Booking Page
A per-tenant scheduling page that defines a service, availability windows, staff, add-ons, questions, and payment settings. Guests pick a time slot on the public `app/book/[slug]/` page. Requires admin approval via a token link before going live. MCP tools: `booking_pages_list`, `booking_pages_get`, `booking_pages_create`, `booking_pages_update`.

### Brain
Short for Company Brain. See **Company Brain** below.

### Bun
The JavaScript runtime and package manager used throughout the project. Always use `bun` instead of `npm` or `yarn`. The lock file is `bun.lock`.

### BYOK (Bring Your Own Key)
A configuration pattern where a tenant supplies their own API credentials (OpenAI key, Stripe keys) instead of using the platform's shared account. BYOK AI keys are encrypted with AES-256-GCM (`lib/crypto/`). BYOK AI key settings are at `app/portal/settings/ai/`.

## C

### clientId
The database primary key identifying a tenant (client). Every data row in a tenant-scoped table carries a `clientId` foreign key. Missing or incorrect `clientId` scoping is the most common tenancy bug. After any data-access change, run `bun test:tenancy` to check for cross-tenant leaks.

### Company Brain
The per-tenant AI knowledge base. Stores notes, decisions, documents (with versioned drafts and acknowledgment tracking), tasks, meetings, people, goals, initiatives, playbooks (with run history), a glossary, a topic tree, an org chart, and relationships. Semantic search is powered by OpenAI embeddings stored in pgvector. The Brain agent classifies intent, plans actions, executes a tool loop, and checks groundedness. 156 MCP tools under the `brain_*` namespace. See [AI overview](./ai-overview.md).

### Custom Post Type
A per-tenant schema extension that lets a client define a new kind of content beyond the built-in `page` / `post` / `blog` types. Each custom type has custom field schemas and an editable Liquid render template. Managed via `post_types_*` MCP tools (13 tools).

---

## D

### Deck (Pitch Deck)
A slide presentation authored in the block editor. Supports AI slide generation, branching decision slides, multi-user collaboration (Yjs), version history, and three public viewer URL schemes. 12 MCP tools under `decks_*`. Public viewers at `app/slides/[slug]/`, `app/pitch-deck/[slug]/`, `app/sites/[domain]/slides/[slug]/`. AB testing is implemented but not wired to the public render paths.

### Decision
A Company Brain record that logs a team decision with context, rationale, and outcome. Decisions can be superseded or rejected. MCP tools: `brain_decisions_list`, `brain_decisions_get`, `brain_decisions_create`, `brain_decisions_update`, `brain_decisions_reject`, `brain_decisions_supersede`.

### Drizzle ORM
The SQL query builder and schema manager used throughout the project. Schema modules live in `lib/db/schema/`. Migrations are generated via `bun run db:generate` and applied via `bun run db:migrate`. Never hand-edit files in `drizzle/*.sql`.

---

## E

### Embedding
A high-dimensional vector representation of text, generated by the OpenAI Embeddings API and stored in Postgres via pgvector. Used by Company Brain for semantic similarity search. The embedding pipeline is async — embeddings may lag note creation by a short interval.

### Entitlement / Bundle
A billing concept that gates access to a platform module or feature. Tenants subscribe to individual modules à-la-carte or in bundles. Entitlement checks live in `lib/billing/`. See [AI overview](./ai-overview.md) for the billing module overview.

---

## G

### Glossary (Brain)
A Company Brain sub-feature that stores canonical term definitions for a tenant's domain vocabulary. Supports bulk import and lookup. MCP tools: `brain_glossary_list`, `brain_glossary_get`, `brain_glossary_create`, `brain_glossary_update`, `brain_glossary_delete`, `brain_glossary_lookup`, `brain_glossary_bulk_import`.

## I

### Initiative
A Company Brain record representing a project or strategic effort. Initiatives can link to other Brain entities and can be opened or closed. MCP tools: `brain_initiatives_list`, `brain_initiatives_get`, `brain_initiatives_create`, `brain_initiatives_update`, `brain_initiatives_link`, `brain_initiatives_links`, `brain_initiatives_unlink`, `brain_initiatives_close`, `brain_initiatives_reopen`.

## M

### MCP (Model Context Protocol)
The protocol by which AI clients (e.g. Claude.ai) call platform capabilities as structured tools. The platform's MCP endpoint is `POST /api/mcp` (Streamable HTTP). 450 tools available under the `*` wildcard scope. See [tool reference](./tool-reference.md) and [AI overview](./ai-overview.md).

### MCP Approval Queue
The set of pending content changes created by MCP write tools and waiting for human approval before going live. Each pending change has an approval link. Portal staff review the queue at `app/portal/approvals/`. See **Approval Link**.

### MCP Tool
A named, schema-validated callable function exposed over the MCP endpoint. Every tool has a scope guard (`hasScope(ctx.scopes, ...)`) that enforces tenant isolation. A missing scope guard is a tenancy bug and will fail the baseline test. See `lib/mcp/` and [tool reference](./tool-reference.md).

### Module Subscription
A Stripe-backed per-tenant subscription to a specific platform capability (e.g. email campaigns, Company Brain, storefront). Tenants are billed à-la-carte or in bundles with volume discounts and per-seat pricing.

## N

### NextAuth
The authentication library (v5 beta) used for session management. Strategy: JWT stored in an httpOnly cookie, 7-day maxAge with 1-day idle refresh. Providers: Credentials (bcrypt + optional TOTP) and Google OAuth. See [AI overview](./ai-overview.md) → Auth section.

## O

### OAuth 2.1 Server
The platform's own authorization server at `lib/oauth/server.ts`. Issues scoped `sd_oauth_` tokens to MCP clients via the standard auth-code flow with PKCE (RFC 7636) and resource indicators (RFC 8707). The consent screen is at `app/oauth/authorize/`. Admin OAuth client management at `app/admin/oauth-clients/`.

### Org Unit
A Company Brain entity representing a team or organizational division. Supports a tree structure (parent/child), members, and primary assignment. MCP tools: `brain_org_units_*` (12 tools).

## P

### pgvector
A Postgres extension (`vector` type) required on every database. Used to store and query OpenAI embedding vectors for Company Brain semantic search. The extension must be enabled before running migrations — `lib/db/schema/` will fail otherwise.

### Playbook
A Company Brain entity that defines a repeatable multi-step process. A playbook has ordered steps; a run tracks progress through those steps. MCP tools: `brain_playbooks_*` (10 tools) and `brain_playbook_runs_*` / `brain_playbook_run_steps_*` (7 tools). Portal UI: `app/portal/brain/playbooks/` and `/playbook-runs/`.

### Plugin
A separately-deployed Next.js application that embeds inside the portal under `/portal/apps/<slug>/` via an HMAC-JWT proxy. Plugins communicate through a 60-second JWT TTL (do not extend without threat-model review). The first live plugin is Content Tools. See `app/portal/apps/` and `app/api/plugin-callback/`.

### Portal (route tree)
The per-tenant client UI at `app/portal/**`. Every portal route resolves the active tenant via the site-resolver middleware and requires an authenticated session. Distinct from the admin panel and the public site. See **Site Resolver**.

### Post
A content record in the CMS. A post's body is a JSON block tree (`{ blocks: Block[], version: '1.0' }`). Posts belong to a site (via `siteId`) and have a post type (page, blog post, or custom type). MCP tools: `posts_list`, `posts_get`, `posts_create`, `posts_update`, `posts_delete`, `posts_fork`, `posts_upload_html`, `posts_upload_html_zip`, `posts_list_revisions`, `posts_set_taxonomies`.

### Post Type
The kind of content a post represents. Built-in types include standard pages and blog posts. Tenants can define custom post types with their own field schemas and render templates via `post_types_*` MCP tools. Not to be confused with WordPress post types — these are stored as JSON schemas, not PHP classes.

### Print Designer
A Fabric.js canvas-based embellishment tool where storefront customers design text, icons, and images on product mockups. **Status: fate decision (invest / defer / cut) is explicitly open.** No service entitlement gate; always on when `products.designable = true`. Two public routes coexist (`/designer/` and `/design/`) pointing at different DB schema tables. No MCP tools.

### Public Site (route tree)
Per-tenant public-facing pages at `app/sites/[domain]/[[...slug]]/` and short-URL handler at `app/s/[slug]/`. Rendered server-side; block content is fetched from Postgres and rendered by the block renderer. No auth required for public visitors.

## R

### RAG (Retrieval-Augmented Generation)
The architecture pattern used by Company Brain: user questions are answered by first retrieving semantically similar documents (via pgvector similarity search on OpenAI embeddings) and then passing those chunks to an LLM as context. The Brain agent adds intent classification, planning, and groundedness checks on top of basic RAG.

### Review Item
A Company Brain inbound email or content item that has been routed to a human review queue (at `app/portal/brain/review/`) for approval, rejection, or action. MCP tools: `brain_get_review_item`, `brain_list_review_items`, `brain_approve_review_item`, `brain_reject_review_item`, `brain_review_items_list_for_reviewer`, `brain_review_items_suggest_reviewer`.

## S

### Scope Guard
A per-tool access check inside each MCP tool registrar: `hasScope(ctx.scopes, '<domain>:<permission>')`. A scope guard confirms the caller's token includes the required scope before the handler executes. Missing a scope guard on a new tool is a tenancy/security bug and will be caught by the baseline test. ~50 named scopes exist; `*` grants all.

### Site
A per-tenant website record (`clientWebsites` table). A site has a domain, branding, navigation, custom CSS/JS, and is the parent of all posts/pages for that website. Identified by `siteId`. A single tenant can have multiple sites. MCP tools: `sites_list`, `sites_get`, `sites_update`, `sites_get_custom_code`, `sites_update_custom_code`, `sites_publish_custom_code`, `sites_create`.

### Site Resolver
Middleware at `lib/active-client.ts` that resolves which tenant (`clientId`) and site (`siteId`) own a given request. Portal routes use session + explicit `siteId` param; public routes use the `Host` header. All data-access calls in portal API routes must use the resolved `clientId` to prevent cross-tenant leaks.

### siteId
The database primary key identifying a specific website within a tenant. Rows scoped to a particular site carry both `clientId` and `siteId`. Some data is tenant-wide (`clientId` only); some is site-specific (`siteId` required).

### Sprint
A time-boxed iteration in the project management module. Kanban cards can be assigned to a sprint. MCP tools: `sprints_list`, `sprints_create`, `sprints_update`, `sprints_delete`. The `kanban_propose_sprint` tool generates a sprint proposal from backlog items.

### Survey
A multi-page, multi-field form with branching logic, scoring, A/B variants, CRM auto-routing, post-submission email sequences, file upload, and a public recommendation engine. Public renderer at `app/s/[slug]/`. MCP tools: `surveys_list`, `surveys_get`, `surveys_create`, `surveys_update`, `surveys_fork`, `surveys_list_responses`.

## T

### Tenant
A single client organization provisioned in the platform. Each tenant has a `clientId`, one or more portal users, one or more sites, and its own isolated data rows. The agency admin manages all tenants from `app/admin/`. In code, "client" and "tenant" are used interchangeably.

### Tenant Isolation
The invariant that one tenant can never read or write another tenant's data. Enforced by requiring `clientId` (and sometimes `siteId`) on every data-access call and scope guard on every MCP tool. Regression gate: `bun test:tenancy`. See `lib/db/CLAUDE.md`.

### Topic
A Company Brain taxonomy node for organizing knowledge. Topics form a tree structure; notes, documents, and other Brain entities can be tagged with topics. MCP tools: `brain_topics_list`, `brain_topics_get`, `brain_topics_create`, `brain_topics_update`, `brain_topics_delete`, `brain_topics_tree`, `brain_topics_attach`, `brain_topics_detach`, `brain_topics_merge`, `brain_topics_move`, `brain_topics_entities`, `brain_topics_import_from_tags`.

### TOTP / MFA
Time-based One-Time Password multi-factor authentication. Shipped 2026-06-26. Users enroll via `app/portal/settings/security/`; the TOTP field appears at login. Implementation: `lib/totp.ts`; `mfaEnabled` + `totpSecret` columns on the users table. Disable requires password re-verification. Fail-closed, no enumeration.

### Trigger Link
A tracked short link (`app/portal/automations/trigger-links/`) that fires an automation rule when clicked. Used for click-based re-engagement flows. The bridge from link click to `contactFieldKey` update is forward-looking, not yet wired.

## V

### Visual Editor
The block-based WYSIWYG page builder at `app/portal/websites/[siteId]/posts/[postId]/edit/`. The portal shell owns block state; a sandboxed iframe loads the live public-site renderer. The two frames communicate over a typed postMessage protocol. Supports drag-and-drop reorder, breakpoint previews, inline text editing, undo/redo, multi-user collaboration presence (Yjs), and contextual AI restyle. No dedicated MCP tools — content is written via `posts_update`. See `components/portal/visual-editor/CLAUDE.md`.

### Visual Workflow
A durable, multi-step automation built on a ReactFlow node canvas. Runs are persisted in Postgres with a queue drainer, exponential-backoff retries, dead-letter handling, and a Retry button in the run history UI. Distinct from automation rules (which are stateless and one-shot). **Status:** shipped to `dev` branch 2026-06-25, pending staging migration before main merge. No MCP tools yet. See `app/portal/automations/workflows/`.

### Voice Assistant
An OpenAI Realtime API voice assistant with CRM and Brain tools and meeting-mode transcript ingestion. **Status: built but dormant.** The widget is not mounted in the portal layout; environment variables are not documented; no E2E tests exist. Do not depend on this feature.

---

## W

### Webhook (outbound)
Per-project outbound HTTP callbacks fired on kanban events. SSRF-guarded via `lib/ssrf-guard.ts`. Known gaps: no retry logic, no signing secret, no delivery log.

### Whoami
An unscoped MCP tool that returns the authenticated caller's identity (tenant, user, scopes). Useful for agent bootstrap. MCP tool: `whoami`.

---

## Y

### Yjs / Collaboration
The CRDT library used for real-time multi-user collaboration in the visual editor and pitch deck editor. Requires a standalone WebSocket server (Railway-hosted) separate from the Next.js app. The Yjs document is the source of truth for live-editing sessions; changes are flushed to Postgres on save.

### [public/chat-widget] Widget iframe postMessage to parent window
- lens: input-abuse · type: coverage-gap · verdict: unverified-lowsev
- at: `app/widget/chat/chat-bootstrap.tsx:52-60`
- evidence: The postParent() function at line 52 calls window.parent.postMessage(message, '*') — the wildcard targetOrigin means the resize/close messages are delivered to any parent regardless of origin. An attacker who embeds the widget iframe in their own page receives sd-chat:resize and sd-chat:close frames, leaking the fact that a user is interacting with the widget. This is low severity since the data in these frames (expanded:boolean, width/height) is not sensitive, but the wildcard is a best-practice violation and is untested.
- proposed: Test that the widget iframe emits sd-chat:resize with a predictable structure and that the targetOrigin is locked to the known embedding origin (or at minimum that the message contents contain no sensitive PII/token data). For a pure coverage-gap: verify the widget iframe page loads, opens, and sends the expected postMessage types.

### [sites/sites-cms] template-preview route — /sites/[domain]/template-preview/[typeId]
- lens: authz-bypass · type: coverage-gap · verdict: confirmed
- at: `app/sites/[domain]/template-preview/[typeId]/page.tsx:31-48`
- evidence: The route gates on _edit=true plus a preview token or session. Without _edit=true it returns a soft 'sign in to preview' message, not a 404/401. A visitor who omits _edit but supplies a valid typeId can confirm the domain exists. More importantly: the postType query at line 53-59 uses or(eq(postTypes.websiteId, site.id), isNull(postTypes.websiteId)) which would let a legitimate user of Site A request typeId belonging to Site B (a globally-scoped built-in with isNull websiteId), not a leak per se but unverified. Zero E2E tests cover this route for any case (auth gate, cross-site typeId, unauthenticated access).
- proposed: E2E: navigate to /sites/domain/template-preview/1 without _edit=true and assert a 200 response with the 'sign in' message (not a block tree). Navigate with ?_edit=true&_token=<invalid> and assert the same 'sign in' message. Navigate with a valid site-wide token and assert the block renderer output is served.
- skeptic: The soft auth gate at lines 41-48 is confirmed: without _edit=true the route returns a friendly message rather than a 404/401. Zero E2E test coverage is confirmed — no test file references template-preview. However, the cross-site typeId claim is refuted: the query `or(eq(postTypes.websiteId, site.id), isNull(postTypes.websiteId))` correctly scopes to the current site's own types plus intentionally global built-ins (isNull websiteId); Site B's private types cannot be fetched through Site A's domain. No data is exposed through the unauthenticated code path (only a text message), and the route is a developer tool (iframe target for the visual editor), so the severity is low rather than high.

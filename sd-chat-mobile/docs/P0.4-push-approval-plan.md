# P0.4 — Push notifications + lock-screen approval

_Implementation plan. Completes the P0 loop: a staged AI write pushes to the owner's phone; they approve from the notification._

## Goal
When an AI-authored write is staged (P0.3 → `stageOrApply` → `notifyApprovers`), send an **Expo push notification** to the workspace owners/admins. The notification carries **Approve / Reject** actions that call the now-bearer-aware approvals API (#41) — so the write can be approved without opening the app (or with a single tap).

This is the last unstarted P0 piece and the one that makes the approval loop feel real on mobile.

## ⚠️ Hard prerequisites (gates full verification)
Unlike P0.1–P0.3, this **cannot be fully verified in this environment**:
- **Dev build required.** Expo Go dropped remote push in SDK 53+; SDK 56 needs an **EAS development build** on a real device (or a configured simulator) to receive remote push.
- **Push credentials.** `getExpoPushTokenAsync()` needs an EAS `projectId` (absent from `app.json`) plus **APNs** (iOS) / **FCM** (Android) credentials configured in EAS — these are account-level and the user must provision them.
- **A physical device** for the actual lock-screen interaction.

What *can* be verified headlessly here: the **portal side** (device-token registry + the send call) by POSTing to the Expo Push API with a token, and the mobile token-registration request shape. The native receive/lock-screen-action path needs the user's device + build.

## Architecture

### Portal (testable headlessly)
1. **Schema** — new `device_push_tokens` table:
   `id, clientId, userId, token (unique, the ExponentPushToken[...]), platform ('ios'|'android'), createdAt, lastSeenAt, revokedAt`.
2. **Register/unregister endpoints** (bearer-aware via `authorizePortal({action:'read'})` — any signed-in member can register their own device):
   - `POST /api/portal/devices` `{ token, platform }` → upsert (token unique; rebind to current user/client).
   - `DELETE /api/portal/devices` `{ token }` → revoke (called on sign-out).
3. **Send hook** — in `stageOrApply` (`lib/mcp/pending-changes.ts`), the block that already calls `notifyApprovers` returns the recipient userIds. After it resolves, look up those users' active `device_push_tokens` and POST to the **Expo Push API** (`https://exp.host/--/api/v2/push/send`) with:
   ```
   { to: token, title: 'Approval needed', body: summary,
     data: { kind: 'mcp_approval', pendingId }, categoryId: 'mcp_approval' }
   ```
   Fire-and-forget, same as the email fan-out; never block the tool response. Wrap in a `lib/push/send.ts` helper (batch ≤100 per Expo call, handle `DeviceNotRegistered` receipts by revoking the token).

### Mobile (needs device/build to verify)
4. **Deps**: `expo-notifications`, `expo-device`. Add EAS `projectId` + iOS `aps-environment` entitlement + Android notification channel to `app.json`.
5. **Register** (`lib/push/register.ts`): on successful sign-in (and app launch when authed), `getExpoPushTokenAsync({ projectId })` → `POST /api/portal/devices`. Unregister on sign-out (before clearing the token). Guard `Device.isDevice` and permission request.
6. **Notification category** with actions, registered at startup:
   `Notifications.setNotificationCategoryAsync('mcp_approval', [{ identifier: 'approve', buttonTitle: 'Approve' }, { identifier: 'reject', buttonTitle: 'Reject', options: { isDestructive: true } }])`.
7. **Response handler** (`addNotificationResponseReceivedListener`): on `approve`/`reject` action (or body tap), read `data.pendingId`, call the bearer-aware `POST /api/portal/approvals/{pendingId}/{approve|reject}` (reuse `lib/api/approvals.ts` mutations). Body tap with no action → deep-link to `/approvals/[id]` (the P0.3 glue screen).

## Security model
- The notification action calls the approvals API with the **bearer token from SecureStore** — only a signed-in user on that device can approve. The API still enforces **owner/admin** (`authorizePortal({action:'admin'})`, #41), so a viewer's tap is rejected server-side.
- Token registry is per-user; sign-out revokes the device token so a reassigned device stops receiving another user's approvals.
- True **headless approve** (action handled without launching the app) is an advanced iOS path — the response listener runs when the app is backgrounded/launched-from-notification. First increment: tap → app handles in foreground/background. Headless background action is a follow-up.

## Phased increments
- **4a — Portal device registry + send** (testable here). Schema + register/unregister endpoints + `lib/push/send.ts` + hook into `stageOrApply`. Verify by registering a dummy token and asserting the Expo Push API call shape (or push to a real device token if the user provides one). **Recommended first.**
- **4b — Mobile registration** (needs build). Deps, `projectId`, register-on-sign-in, unregister-on-sign-out.
- **4c — Notification handling + actions** (needs device). Category, response listener, approve/reject + deep-link.
- **4d — Headless background approve** (advanced, optional).

## Effort
**L–XL (~1 week)**, most of it in 4b–4d native setup + device testing the user must drive. 4a is **M** and landable/verifiable now.

## Risks
- EAS credential setup is account-level and user-owned — blocks 4b+.
- iOS background action limits (4d) may push "approve without opening app" to a follow-up.
- Expo Push receipts must be reaped (`DeviceNotRegistered`) or the token table accumulates dead tokens.

# P0.1 — Streaming tool-calling (portal Phase 4)

_Implementation plan. Unblocks the mobile assistant acting across domains + the approvals loop (P0.2, P0.3)._

## Goal
Make `POST /api/portal/ai/chat/stream` run the **agentic tool loop** while streaming — emitting the `tool_call` / `tool_result` SSE frames the mobile client already parses (`lib/api/chat-stream.ts`) and renders (`components/chat/ToolUseCard.tsx`). Result: a mobile user can ask the assistant to create a deal, move a card, draft an email, etc., and watch it happen token-by-token.

## Where the two routes stand today
| | Non-streaming `/ai/chat/route.ts` | Streaming `/ai/chat/stream/route.ts` |
|---|---|---|
| Auth | NextAuth `auth()` + `authorizePortal` | `resolvePortalFromRequest` (bearer) ✅ keep |
| Tools | `while` loop, `messages.create({ tools: PORTAL_TOOLS })`, `executePortalTool(...)`, caps MAX_LOOPS=8 / MAX_TOOL_CALLS=20 | **none** — single `messages.stream()`, text only |
| Output | JSON `{ reply, toolCalls, tokensUsed }` | SSE `token` → `done` |
| Persistence | user + assistant rows; assistant stores `toolCalls` w/ results; sums tokens across loops | user row upfront; assistant text only; single-call tokens |
| Model | `claude-sonnet-4-6` | `claude-opus-4-7` |
| System prompt | rich (linking + confirmation + capabilities) | terse |

**The task is to graft the left column's loop onto the right column's streaming transport.**

## Core design: stream *each* loop turn, then branch on `stop_reason`
Replace the single `messages.stream()` call with a `while` loop. Each iteration streams one assistant turn; we relay text deltas live, then use the SDK's assembled final message to decide whether to run tools and continue, or finalize.

Key SDK fact: `anthropic.messages.stream({ tools, ... })` returns a `MessageStream` that is both async-iterable (for live `text_delta`s) **and** exposes `await stream.finalMessage()` → the fully-assembled `Message` with parsed `tool_use` blocks, `stop_reason`, and `usage`. Use the iterator for live tokens and `finalMessage()` for the tool decision — do **not** hand-reassemble args from `input_json_delta`.

### Pseudocode (replaces the body of the `ReadableStream.start`)
```ts
let currentMessages = [...anthropicMessages];   // history + new user turn
const allToolCalls = [];                        // {name, input, result} for persistence
let assistantText = '';                         // text of the FINAL turn (persisted)
let inputTokens = 0, outputTokens = 0;          // ACCUMULATED across loop turns
const MAX_LOOPS = 8, MAX_TOOL_CALLS = 20;
let loops = 0, toolCalls = 0;

while (loops < MAX_LOOPS) {
  loops++;
  let turnText = '';
  const sdkStream = anthropic.messages.stream({
    model: MODEL, max_tokens: MAX_TOKENS, system: sysParam,
    tools: PORTAL_TOOLS, messages: currentMessages,
  });

  for await (const ev of sdkStream) {
    if (ev.type === 'content_block_delta' && ev.delta.type === 'text_delta') {
      turnText += ev.delta.text;
      emit({ type: 'token', text: ev.delta.text });   // live relay
    }
  }

  const msg = await sdkStream.finalMessage();
  inputTokens  += msg.usage.input_tokens  ?? 0;
  outputTokens += msg.usage.output_tokens ?? 0;

  if (msg.stop_reason !== 'tool_use') {
    assistantText = turnText;                          // last turn's text is the reply
    break;                                             // → finalize() emits done
  }

  // stop_reason === 'tool_use'
  const useBlocks = msg.content.filter(b => b.type === 'tool_use');
  toolCalls += useBlocks.length;
  if (toolCalls > MAX_TOOL_CALLS) { await finalize('tool_call_cap_exceeded'); return; }

  const results = [];
  for (const b of useBlocks) {
    emit({ type: 'tool_call', tool: b.name, args: b.input, id: b.id });   // ← mobile frame
    const result = await executePortalTool(b.name, b.input, client.id, userId);
    allToolCalls.push({ name: b.name, input: b.input, result });
    emit({ type: 'tool_result', id: b.id, output: result });             // ← mobile frame
    results.push({ type: 'tool_result', tool_use_id: b.id, content: JSON.stringify(result) });
  }

  currentMessages = [...currentMessages,
    { role: 'assistant', content: msg.content },   // includes tool_use blocks
    { role: 'user', content: results },
  ];

  if (loops >= MAX_LOOPS) { assistantText = turnText; break; }   // graceful cap
}

await finalize();   // persists assistantText + allToolCalls, deducts credits, emits done
```
`emit(x)` = `controller.enqueue(encoder.encode(sseFrame(x)))`.

## Concrete changes to `stream/route.ts`
1. **Import** `PORTAL_TOOLS, executePortalTool` from `@/lib/ai/portal-tools`.
2. **Swap the single `messages.stream` call** for the loop above.
3. **Accumulate tokens** across turns (today `inputTokens/outputTokens` are set once from `message_start`/`message_delta`; with `finalMessage()` you read `msg.usage` per turn and `+=`). The existing `finalize()` already deducts on the accumulated totals — just feed it the summed values.
4. **Persist tool calls**: in `finalize()`, add `toolCalls: allToolCalls.length ? allToolCalls : null` to the assistant `aiMessages` insert (mirror the non-streaming route's row shape so the admin conversation viewer + mobile history render identically).
5. **Adopt the rich `SYSTEM_PROMPT`** from `/ai/chat/route.ts` (linking rules + confirmation rules + capability overview), keeping the `cache_control: ephemeral` breakpoint. Extract it to a shared module (`lib/ai/portal-chat-prompt.ts`) so both routes stay in sync — today they will drift.
6. **Caps → error frames, not HTTP status.** Once SSE headers are sent you can't return a 400. On `tool_call_cap_exceeded` / `loop_cap_exceeded`, emit `{ type: 'error', message }` then `done` (finalize handles both). The mobile error path already renders this.
7. **Model**: align with the working agentic loop. The non-streaming loop uses `claude-sonnet-4-6` (fast, tool-reliable). Recommend the stream loop match it rather than `claude-opus-4-7` — tool loops fan out multiple round-trips and latency compounds. Make `MODEL` a single shared const. (Quick-win: both are behind current; revisit vs. Opus 4.8 / Fable 5 once the loop is proven.)

## Decision point — this is where P0.1 meets P0.3 (approvals)
`executePortalTool` **executes writes directly**; the only safeguard today is the prompt-level "summarize → ask for yes → then call" guardrail in the rich system prompt. It does **not** route through `lib/mcp/pending-changes.ts` (that gate lives in the MCP tools `lib/mcp/tools/*`, keyed on the API key's `require_cms_approval`). Mobile keys are minted full-trust (`requireCmsApproval: false`, scopes `['*']`).

So out of the box, a mobile assistant write **commits immediately** and never lands in the approvals inbox. Pick one before shipping:
- **(A) Ship trust-based first** (prompt confirmation only). Simplest; lights up the value loop fastest. The approvals inbox stays external-agent-only for now. ← recommended for P0.1.
- **(B) Make write tools approval-aware.** Teach `executePortalTool` (or a wrapper) to check `require_cms_approval` for the resolved key and, for the destructive tool set enumerated in the system prompt, write a `mcpPendingChanges` row + emit a `tool_result` of `{ status: 'pending_approval', approvalUrl }` instead of committing. This is the real P0.3 and should be its own ticket — don't block P0.1 on it.

Flag this explicitly to the product owner; it's a trust/safety call, not an implementation detail.

## Testing
1. **Unit-ish (curl):** `curl -N` the stream endpoint with a bearer token and a prompt that forces a read tool ("what are my open deals?") → assert `token` + `tool_call`(`list_crm_deals`) + `tool_result` + `done` frames in order.
2. **Multi-tool:** "summarize my pipeline and overdue invoices" → expect ≥2 `tool_call`/`tool_result` pairs across ≥1 loop.
3. **Cap:** craft a prompt that loops → assert graceful `error` + `done`, no hang, no orphaned SSE.
4. **Write (guarded):** "create a contact named Test" → assert the assistant asks to confirm first (no premature `tool_call`), then on "yes" emits the write `tool_call`/`tool_result`.
5. **Persistence:** reopen the conversation → assistant row has `toolCalls` populated; tokens summed across loop turns; credits deducted once on the total.
6. **Disconnect:** abort mid-loop → `finalize()` is idempotent, user message already persisted, no double-deduct.
7. **Device:** drive `app/chat/[id].tsx` end-to-end in the simulator; watch `ToolUseCard` render live.

## Risk / watch-items
- **Latency:** each tool round-trip is a fresh model call; the loop can take many seconds. The streaming UX hides this (tokens + cards keep arriving), but set client-side timeouts generously (the stream client uses `timeout: 0` ✅).
- **Prompt drift:** if the two routes keep separate prompts, tool behavior diverges. Extract to one module (change #5).
- **Tool errors:** `executePortalTool` failures should serialize into the `tool_result` output (so the model can recover/retry) rather than throwing out of the loop. Mirror non-streaming behavior.
- **Credit exhaustion mid-loop:** today credits are checked once upfront. A long tool loop could overspend. Optional follow-up: re-check `hasCredits` between turns and break with an `error` frame.
